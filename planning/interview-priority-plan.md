# 嵌入式学习计划 v3 — 理论×实践×追觅经验融合

> 核心理念：不是"学完再找工作"，而是**用一个完整项目把追觅经验、408 理论、嵌入式实践全部串起来**。
> 这个项目就是你面试时最有力的证据。

---

## 为什么这个思路是对的

面试官最怕两种人：
1. "我学过 STM32" — 但一问项目就卡壳
2. "我在追觅做测试" — 但说不清楚这和嵌入式有什么关系

你有一个别人没有的优势：**你在真实的扫地机器人产品上工作了一年。** 你知道一台扫地机从研发到量产经历了什么、什么模块容易出 bug、产线上什么指标最关键。这些东西如果只是口头描述，面试官听不出价值。但如果你**自己动手复现了一个扫地机的核心子系统**，并且在设计时融入了你在追觅学到的工程思维，面试官听到的就不再是"我在追觅做测试"，而是：

> "我在追觅做了一年扫地机器人系统验证，理解了整机架构和量产质量体系。然后我自己用 STM32 复现了底盘控制子系统——从 MCU 固件到传感器驱动到 Python 测试框架——并且把追觅的测试标准和质量管理方法应用到了自己的项目里。"

这是一个完整的故事闭环：**学理论 → 看产品 → 自己造 → 验证质量 → 讲出来。**

## 核心项目：SmartClean 底盘控制子系统

定位：一个扫地机器人底盘的最小可行系统（Minimum Viable System），包含 MCU 固件 + 传感器 + 电机控制 + 测试框架。

### 系统架构

```
┌─────────────────────────────────────────┐
│         PC (Python 测试框架)              │
│  - 实时传感器数据可视化（matplotlib）      │
│  - 自动化测试脚本（pytest）               │
│  - 数据记录与分析（pandas, CSV）          │
│  - 质量报告生成                           │
└──────────────┬──────────────────────────┘
               │ UART (二进制帧协议)
┌──────────────▼──────────────────────────┐
│      STM32F103 + FreeRTOS                │
│                                          │
│  ┌──────────────────────────────────┐    │
│  │ Task 1: 传感器采集 (100Hz)        │    │
│  │  - VL53L1X ToF (I2C, 悬崖检测)    │    │
│  │  - ADC 电池电压/电流               │    │
│  │  - 碰撞传感器 (GPIO 中断)          │    │
│  ├──────────────────────────────────┤    │
│  │ Task 2: 电机控制 (1kHz PID)       │    │
│  │  - BLDC 六步换向 (TIM1 互补PWM)   │    │
│  │  - Hall 传感器速度反馈             │    │
│  │  - 速度 PI 闭环                    │    │
│  ├──────────────────────────────────┤    │
│  │ Task 3: 通信协议 (50Hz)           │    │
│  │  - 二进制帧打包/解包               │    │
│  │  - UART DMA 发送 + 环形缓冲接收    │    │
│  │  - 帧序号 + CRC16 校验             │    │
│  ├──────────────────────────────────┤    │
│  │ Task 4: 状态机 (10Hz)             │    │
│  │  - 系统状态管理 (IDLE/RUN/ERROR)   │    │
│  │  - 故障检测 (过流/堵转/低电压)     │    │
│  │  - Watchdog 喂狗                  │    │
│  ├──────────────────────────────────┤    │
│  │ Task 5: 数据日志                   │    │
│  │  - UART 调试输出 (printf)          │    │
│  │  - 运行时统计 (CPU 使用率/栈水位)   │    │
│  └──────────────────────────────────┘    │
└──────────────────────────────────────────┘
```

### 为什么选这个项目

| 原因 | 说明 |
|------|------|
| **覆盖面试技能树** | FreeRTOS 多任务 / I2C/SPI/UART / BLDC 电机 / PID / DMA / 环形缓冲区 — 全在 P0 清单里 |
| **追觅经验落地** | FCT 测试框架 → Python 上位机；产线质量指标 → 故障检测状态机；传感器数据分析 → 数据记录 |
| **408 理论映射** | 环形缓冲区=循环队列、FreeRTOS 调度=OS 进程调度、CAN/UART=网络协议栈思维 |
| **有实物证据** | 电机转动视频 + 逻辑分析仪波形 + Python 可视化截图 + 测试报告 |
| **一个项目讲完所有面试问题** | 不用准备 5 个分散的项目，一个项目就能回答所有技术面问题 |

## 四层融合：追觅经验如何在项目中落地

### 层级 1：FCT 经验 → Python 测试框架

| 在追觅见过/做过的事 | 映射到你的项目 |
|-------------------|---------------|
| FCT 自动化工装（pandas/matplotlib 数据分析） | 用 Python 写上位机，实时读取 STM32 串口数据，画传感器曲线 |
| 产线测试 PASS/FAIL 判定 | 写 pytest 自动化测试脚本，自动判断电机转速是否达标、传感器数据是否超界 |
| 测试报告生成 | 每次跑完自动生成 CSV 日志 + matplotlib 图表，附质量判断 |

**面试能讲的故事**：
> "我在追觅做 FCT 的时候，用 Python 写自动化测试工具分析产线数据，效率提升了 3 倍。在做 SmartClean 项目时，我把同样的思路用在了自己的嵌入式系统上——用 pytest + pyserial 写了完整的自动化测试框架，每次固件更新后自动跑 12 个测试用例，包括电机转速精度、传感器数据稳定性、通信帧 CRC 校验。这种'嵌入式工程师写测试'的习惯，大部分纯嵌入式背景的人没有。"

### 层级 2：产品质量思维 → 故障检测状态机

| 在追觅见过/做过的事 | 映射到你的项目 |
|-------------------|---------------|
| 产品可靠性测试（高温老化、跌落测试） | 在代码里加了故障检测：过流保护、堵转检测、低电压报警 |
| 产线不良品分析（为什么这个传感器报错） | 传感器数据超出阈值时自动标记 + 记录日志，方便回溯 |
| 软件版本管理 / 变更追溯 | 固件版本号通过串口协议上报，Python 端校验版本一致性 |

**面试能讲的故事**：
> "我在追觅见过太多量产后的质量问题——很多是嵌入式固件没有考虑边界条件导致的。所以我在自己项目里设计了一个故障检测状态机：过流超过 2A 连续 100ms → 停止电机；电池电压低于 10.5V → 报警；10 秒没收到心跳帧 → 进入安全模式。这些逻辑来自我对扫地机量产问题的理解——不是理论，是真遇到过的问题。"

### 层级 3：跨部门联调经验 → 通信协议设计

| 在追觅见过/做过的事 | 映射到你的项目 |
|-------------------|---------------|
| 嵌入式 ↔ 上位机联调时经常协议对不上 | 设计了带帧头/帧长/CRC16 的二进制协议，Python 端和 STM32 端都有单元测试 |
| 理解扫地机的传感器到算法数据流 | 在自己的项目里复现了传感器→MCU→上位机的完整数据链路 |

**面试能讲的故事**：
> "在追觅我和嵌入式团队联调的时候，最头疼的就是通信协议——帧格式对不上、字节序搞反、CRC 算错。所以我在自己项目里从零设计了一个二进制帧协议：2 字节帧头 + 1 字节长度 + 1 字节命令 + N 字节数据 + 2 字节 CRC16。STM32 端用 union 做序列化/反序列化，Python 端用 struct 解包。协议问题在开发阶段就被单元测试覆盖了——这是我从产线调试的教训中学到的。"

### 层级 4：408 理论 → 工程实践

| 408 理论 | 在你的项目中怎么用的 |
|----------|-------------------|
| 循环队列 | UART RX 环形缓冲区，DMA 写入 + 任务读取，经典的生产者-消费者 |
| 双向链表 | 理解 FreeRTOS 内核就绪任务链表的实现 |
| 进程调度 | FreeRTOS 按优先级抢占式调度，你的 5 个任务就是这个模型的具体实例 |
| PV 操作 | 传感器任务和通信任务之间的数据传递用队列（生产者-消费者同步） |
| 死锁避免 | 你只有一个互斥量（UART 发送），不存在死锁——这是"最小化共享资源"的设计原则 |
| TCP 可靠传输思想 | CRC16 校验 + 帧序号 + 超时重传，和 TCP 的校验和/序号/超时重传逻辑一致 |

## 修正后的学习优先级

原来的优先级按"面试考频"排，现在改成按"**项目推进顺序**"排——因为项目本身就是最好的面试准备。

### Phase 0：项目骨架搭建（6 月下旬，1 周）

**目标**：STM32 FreeRTOS 5 任务骨架能编译运行。

| 任务 | 内容 | 追觅经验融合点 |
|------|------|-------------|
| 0.1 | FreeRTOS 5 任务创建 + 优先级分配 | 按你的系统设计经验决定任务优先级 |
| 0.2 | UART DMA + 环形缓冲区接收 | 回忆追觅联调时的协议问题 |
| 0.3 | 二进制帧协议定义（帧头/长度/CRC16） | 设计时就写单元测试 |

**验证**：Python 端发"ping"，STM32 回"pong"。

### Phase 1：核心驱动（7 月，4 周）

**目标**：4 个核心模块全部调通，每个都附带逻辑分析仪验证。

| 周 | 模块 | 具体内容 |
|----|------|---------|
| W1 | VL53L1X I2C 驱动 | 裸机寄存器级读写，逻辑分析仪抓 I2C 波形 |
| W2 | BLDC 六步换向 | TIM1 互补 PWM，Hall 传感器读取，电机转起来 |
| W3 | ADC 电压/电流采集 | DMA 多通道采集，Python 上位机画实时电压曲线 |
| W4 | 碰撞传感器 + GPIO 中断 | EXTI 配置，中断回调通知状态机任务 |

**追觅经验融合**：每个模块写好再集成进 RTOS，先单元测试再做集成——这是追觅的测试流程教会你的。

### Phase 2：闭环控制 + 故障检测（8 月上旬，2 周）

| 周 | 模块 | 追觅经验融合点 |
|----|------|-------------|
| W5 | BLDC 速度 PID 闭环 | 调参过程录视频，展示 PID 参数变化对响应的影响 |
| W6 | 故障检测状态机 | 故意制造过流/堵转/低电压场景，验证检测逻辑 |

### Phase 3：Python 测试框架（8 月中旬，1 周）

| 任务 | 内容 |
|------|------|
| 3.1 | pyserial 实时读取 + matplotlib 多通道波形显示 |
| 3.2 | pytest 测试用例（电机转速达标、传感器读数稳定、CRC 校验通过） |
| 3.3 | CSV 日志记录 + 测试报告自动生成 |

**这是你的杀手锏**：绝大部分嵌入式候选人不会写测试框架。你在追觅 FCT 做的就是这个。

### Phase 4：系统联调 + 简历打磨（8 月下旬，2 周）

| 任务 | 内容 |
|------|------|
| 4.1 | 全系统稳定性测试（连续运行 30 分钟，零帧丢失/零故障误报） |
| 4.2 | 录制项目演示视频（电机 PID 调参对比 + 逻辑分析仪波形 + Python 上位机画面） |
| 4.3 | 写项目 README（架构图 + 代码结构 + 测试报告 + 性能数据） |
| 4.4 | 简历 v2：以 SmartClean 项目为中心重写简历 |

## 面试时怎么讲这个项目

不需要把 5 个 task 全讲一遍。准备一个大版本（10 分钟深入）和一个小版本（3 分钟概述）。

### 3 分钟版本（面试开场）

> "我做了一个扫地机器人底盘控制子系统，叫 SmartClean。架构是 STM32 + FreeRTOS 5 任务并行——传感器采集、电机 PID 控制、UART 通信协议、故障检测状态机、数据日志。同时用 Python 写了一个完整的测试框架，pytest 自动化跑 12 个测试用例。这个项目的设计思路来自我在追觅一年做扫地机器人系统验证的经验——我知道什么模块容易出 bug、产线上什么指标最关键，所以我从开发阶段就融入了量产质量思维。"

### 深入追问怎么回答（从你追觅经验引出）

| 面试官问 | 你的回答（追觅经验锚点） |
|---------|---------------------|
| "为什么加故障检测状态机？" | "在追觅我见过量产后的过流烧板、堵转冒烟问题——大部分是固件没做边界保护。所以我给自己的项目加了。" |
| "为什么用 Python 写测试框架？" | "追觅的 FCT 工装就是 Python 写的。我习惯了'写完代码先写测试'的流程，效率比反复手动验证高得多。" |
| "你的通信协议怎么设计的？" | "参考了追觅嵌入式同事的设计思路——帧头/长度/CRC，同时在 Python 和 C 两边都写了单元测试，避免联调时对不上。" |
| "这个项目的难点在哪？" | "BLDC 换相噪声——换相瞬间 ADC 采样会被电流尖峰干扰。我用逻辑分析仪抓到噪声窗口，改到 PWM OFF 期间采样，噪声从 ±30% 降到 ±3%。这个排查思路和我在追觅分析产线不良品时一样。" |

## 修正后的优先级总结

| 优先级 | 原来（按面试考频） | 现在（按项目推进） |
|--------|------------------|------------------|
| 🔴 P0 | C 语言 / STM32 / FreeRTOS / 协议理论 | FreeRTOS 骨架 → 传感器驱动 → 电机控制 → 通信协议（全部融在一个项目里） |
| 🟡 P1 | BLDC 实物 / 简历打磨 | Python 测试框架 / 故障检测状态机 / 追觅经验包装 |
| 🟢 P2 | MQTT / Python 碎片 | 项目演示视频 / README 文档 / GitHub 活跃度 |
| ❌ 砍掉 | Linux 驱动 / RISC-V / TinyML | 不变 |

**核心变化**：不再分头学 C 语言、STM32、FreeRTOS，而是在 SmartClean 项目推进的过程中**遇到什么补什么**。项目跑通了，面试能讲的所有 P0 和 P1 技能自然就覆盖了。
